# Disk Usage Analyzer

Утилита для анализа использования дискового пространства, демонстрирующая преимущества паттерна **Компоновщик (Composite)**.

## Описание программы

Программа сканирует указанную директорию и предоставляет отчет о размерах файлов и папок: древовидную структуру, список самых больших элементов и сводную статистику.

## Паттерн Компоновщик в программе

### Роли классов

| Класс | Роль | Описание |
|-------|------|----------|
| FileSystemNode | Компонент | Абстрактный базовый класс, определяющий единый интерфейс |
| File | Лист (Leaf) | Представляет файл. Не может содержать детей |
| Directory | Композит (Composite) | Представляет папку. Может содержать другие Directory или File |

### Как это работает

- FileSystemNode определяет единый интерфейс для всех элементов (getSize, print)
- File - лист, getSize возвращает размер файла
- Directory - композит, может содержать детей. getSize рекурсивно суммирует размеры всех детей
- Клиентский код работает через единый интерфейс, не заботясь о типе узла

Пример кода с паттерном:
```
// Единообразный интерфейс
virtual uint64_t getSize() const = 0;
virtual void print(...) const = 0;

// В Directory:
uint64_t getSize() const override {
    if (!sizeCalculated_) {
        cachedSize_ = 0;
        for (const auto& child : children_) {
            cachedSize_ += child->getSize();  // Не важно, файл или папка!
        }
    }
    return cachedSize_;
}

// В main:
root->print(...);  // Один вызов, всё само работает
``` 


## Преимущества использования паттерна

1. Единообразие - файлы и папки обрабатываются одинаково через общий интерфейс
2. Рекурсивность - операции автоматически применяются ко всей иерархии
3. Расширяемость - новые типы узлов добавляются без изменения существующего кода
4. Отсутствие проверок типа - полиморфизм заменяет каскады if/else
5. Упрощение клиентского кода - не нужно знать, с каким именно узлом идет работа

## Недостатки программы без паттерна

Для сравнения была написана альтернативная версия без использования Composite. Её недостатки:

1. Проблема хранения - как хранить список детей, если дети могут быть разными типами? Приходится держать ненужные поля (у файла - пустой вектор детей, трата памяти)

2. Проверки типа - везде нужны проверки if (node.isDirectory) { ... } else { ... }, что загромождает код

3. Дублирование кода - функции для файлов и папок похожи, но требуют раздельной логики

4. Опасные состояния - у файла поле children не используется, но существует. У папки поле size дублирует сумму детей

5. Сложность добавления новых типов - при добавлении символьной ссылки придется модифицировать все функции с проверками типа

Пример кода без паттерна:
```
// Нет единого интерфейса, нужны проверки
if (node.isDirectory) {
    // Обработка папки
} else {
    // Обработка файла
}

// В printTree:
if (node.isDirectory) {
    std::cout << node.name << "/ (" << formatSize(node.size) << ")";
} else {
    std::cout << node.name << " (" << formatSize(node.size) << ")";
}
```

## Сравнение двух подходов

| Критерий                  | С паттерном               | Без паттерна              |
|---------------------------|---------------------------|---------------------------|
| Проверки типа             | 0 (полиморфизм)           | 10+ (if isDirectory)      |
| Добавление нового типа    | Новый класс               | Изменение всех функций    |
| Расширяемость             | Высокая                   | Низкая                    |
| Количество строк кода     | около 450                 | около 500                 |
